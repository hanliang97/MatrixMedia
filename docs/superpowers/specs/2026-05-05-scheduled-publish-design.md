# 定时发布设计

## 目标

为本地视频发布增加一次性定时发布能力。GUI 与 CLI 都可以创建定时发布任务，任务创建后立即进入发布历史；到指定时间后由主进程执行现有发布流程。

本设计只支持明确的年月日时分秒时间点，不支持每日、每周、每月等循环规则。

## 用户行为

GUI 本地发布弹窗允许用户选择立即发布或定时发布。定时发布需要填写完整时间，精确到秒，例如 `2026-05-05 20:30:00`。

CLI 使用 `--publish-at "YYYY-MM-DD HH:mm:ss"` 创建定时发布任务。CLI 入参必须包含实际发布需要的数据，包括平台、账号、视频路径、标题、短标题、标签和地址等字段。AI 调用 CLI 时不能只创建空的时间占位任务。

所有任务创建后都写入发布历史：

- `scheduled`：定时任务已创建，等待执行。
- `publishing`：到点后开始执行。
- `success`：发布成功。
- `failed`：到点执行但发布失败。
- `expired`：应用关闭或 CLI 进程未运行导致错过发布时间，启动时发现任务已过期。

发布历史中，`failed` 与 `expired` 都允许用户点击重新发布。重新发布复用历史记录中的真实视频文本、账号、平台和文件路径。

## 架构

新增主进程调度器，负责读取、启动和更新定时任务。调度器复用现有 `runPuppeteerTask`，不新建发布自动化实现。

数据层仍以现有 `pushData` 为发布历史来源。定时发布任务创建时写入一条完整 `pushData` 记录，并携带调度字段，例如：

- `publishStatus: "scheduled"`
- `scheduledPublishAt`: 时间戳
- `scheduledPublishAtText`: 用户输入的时间字符串
- `scheduledTask: true`
- `lastPublishMessage: "等待定时发布"`

应用启动时调度器读取 `pushData` 中未完成的定时任务：

- 未到时间：注册定时器。
- 已过时间：标记为 `expired`，消息为 `任务过期，未执行发布`。

运行过程中 GUI 或 CLI 创建定时任务后，调度器立即刷新或注册对应任务。

## 数据流

GUI：

1. 用户填写本地视频发布内容。
2. 选择账号平台。
3. 选择立即发布或填写定时发布时间。
4. 定时发布时写入完整历史记录，状态为 `scheduled`。
5. 主进程调度器到点后执行该任务。

CLI：

1. `cli publish` 解析普通发布参数。
2. 如果存在 `--publish-at`，校验时间格式和未来时间。
3. 写入完整历史记录，状态为 `scheduled`，输出定时任务创建结果后退出。
4. 后续由应用主进程启动后的调度器负责执行。

到点执行：

1. 调度器将历史记录更新为 `publishing`。
2. 组装现有 `runPuppeteerTask` 载荷。
3. 复用现有发布结果回写逻辑，将状态更新为 `success` 或 `failed`。

## 错误处理

时间格式无效时，GUI 提示用户修正；CLI 返回参数错误退出码 `2`。

CLI 创建定时任务时，如果 `--publish-at` 早于当前时间，返回参数错误，不创建新任务。

应用启动扫描历史记录时，如果发现 `scheduledPublishAt` 已早于当前时间，则标记为 `expired`，不自动补发，避免用户不知情地发布过期内容。

到点执行失败时保持发布历史记录，状态为 `failed`，消息写入失败原因。

如果定时任务引用的视频文件已不存在，到点后标记为 `failed`，消息说明本地视频文件不存在。

## 测试

重点验证：

- GUI 创建定时任务后，发布历史立即出现 `scheduled` 记录。
- CLI 使用 `--publish-at` 后，发布历史立即出现完整记录。
- 应用启动时，未到点任务被调度。
- 应用启动时，已过期任务被标记为 `expired`。
- 到点任务复用现有发布流程，并写回成功或失败状态。
- 发布历史中 `failed` 与 `expired` 可以重新发布。
- CLI Skill 文档能让 AI 正确生成带真实视频文本的 `--publish-at` 命令。
